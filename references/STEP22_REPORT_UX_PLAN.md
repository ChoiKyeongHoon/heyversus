# Step 22 – 신고 UX(/polls 리스트 노출) 결정 기록

## 배경

- 현재 신고(Report)는 `/poll/[id]` 상세 화면에서만 가능하며(`ReportButton`), `/polls` 리스트 카드에서는 제공하지 않습니다.
- 리스트 카드에 신고 버튼을 추가하면 “상세 진입 없이 즉시 신고”가 가능해져 접근성이 올라가지만, 동시에 악용(대량 신고 트롤링) 임계치도 낮아집니다.

## 결론(1번안)

- **현 상태 유지**: 신고 버튼은 **`/poll/[id]` 상세에서만** 제공하고, `/polls` 리스트에는 추가하지 않습니다.

## 근거

- `/polls`는 한 화면에 여러 투표가 노출되어 “보이는 대로 누르는” 형태의 트롤링이 상대적으로 쉬움.
- 현재 신고 생성은 로그인만 요구하며, **사용자별 레이트 리밋/중복 신고 방지 같은 서버 측 방어가 선행되어 있지 않음**.
- 운영 관점에서 허위 신고가 급증하면 관리자 처리 비용이 커지고, 실제 문제 신고의 신뢰도도 하락할 수 있음.

## 범위(이번 결정에 포함/미포함)

- 포함: “리스트 신고 버튼 미제공” 방향을 문서/로드맵에 기록.
- 미포함: UI 변경, API/RPC 변경, DB 스키마 변경.

## 향후 재검토 조건(리스트 신고 UX를 다시 논의할 때)

리스트에 신고 버튼을 노출하려면 최소한 아래 방어책을 먼저 적용한 뒤 재평가합니다.

### 서버 측 방어(우선)

- **중복 방지**: 동일 사용자(`reporter_user_id`)가 동일 투표(`poll_id`)에 대해 일정 기간 내 반복 신고를 제한(예: (user, poll) 단위 고정 쿨다운).
- **레이트 리밋**: 사용자별 신고 생성 빈도 제한(예: 10회/일, 3회/5분 등). 임계값/시간창은 운영 데이터 기반으로 조정.
- **유효성 강화**: 과도하게 짧은 상세 내용/의미 없는 입력을 제한(선택). 사유(`reason_code`) 기반 가중치/추가 확인 플로우도 검토.

### UX 완화(선택)

- 리스트에서는 “신고”를 **즉시 노출하지 않고** `⋯` 메뉴 내부로 넣거나, 확인 단계(“정말 신고하시겠어요?”)를 강화해 오발·트롤링을 완화.
- 신고 유도는 상세에서 충분히 제공하되, 리스트에서는 **공유/즐겨찾기** 중심 액션만 유지.

## 운영/모니터링 제안(재검토 시)

- 지표: 신고 생성량(일/주), 사용자당 신고 수, 투표당 신고 수, `dismissed` 비율, 신고 처리 리드타임.
- 알림: 특정 투표에 신고가 단기간 급증하는 경우 관리자 알림(옵션).
